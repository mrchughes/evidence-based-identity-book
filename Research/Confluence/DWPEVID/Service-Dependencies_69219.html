Core Service Dependency Graph

The evidence-based identity platform implements a carefully orchestrated dependency structure that ensures proper processing flow while maintaining service autonomy and fault tolerance. Dependencies follow the principle that evidence flows from ingestion through identity resolution and semantic translation before reaching award calculation, with cross-cutting services like audit and notification supporting all primary workflows.

Evidence Ingestion Dependencies

Evidence Ingestion Service operates as the primary entry point with minimal upstream dependencies, requiring only the External Integration Gateway for partner submissions and direct citizen/caseworker inputs through presentation layer interfaces. This positioning ensures that evidence can be captured and preserved even when downstream services experience problems.

Downstream Dependencies:

  • Identity Resolution Service: Required for correlating new evidence to existing identity clusters or creating new clusters when no matches are found

  • Audit & Compliance Service: Essential for comprehensive logging of all evidence handling and processing decisions

  • Kafka Event Stream: Critical for publishing evidence events that trigger downstream processing workflows

Failure Handling:

  • Evidence documents are persisted immediately upon receipt, ensuring no data loss if downstream services are unavailable

  • Processing status tracking enables recovery and retry of failed operations without requiring evidence resubmission

  • Queue-based processing allows evidence ingestion to continue even when identity resolution is temporarily unavailable

Data Dependencies:

  • Primary: Triple store for assertion persistence, document store for binary file storage

  • Cache: Redis for processing status and frequently accessed assertion metadata

  • Events: Kafka topics for evidence-received, processing-completed, and error events

Identity Resolution Dependencies

Identity Resolution Service depends on Evidence Ingestion for trigger events but operates with sophisticated caching and batch processing capabilities that reduce real-time dependency requirements. The service can process evidence correlation even when other services are unavailable, with results cached for downstream consumption when services recover.

Upstream Dependencies:

  • Evidence Ingestion Service: Provides trigger events for new evidence requiring identity correlation

  • Semantic Translation Service: Supplies cross-context identity verification when correlating evidence from different policy domains

Downstream Dependencies:

  • Award Calculation Service: Consumes identity confidence scores and cluster information for entitlement decisions

  • Investigation Queue Service: Receives cases when identity confidence drops below required thresholds

  • Audit & Compliance Service: Logs all identity correlation decisions and confidence score changes

Machine Learning Dependencies:

  • Model Training Pipeline: Offline dependency for updating correlation algorithms based on verified identity decisions

  • Feature Store: Redis-cached features for real-time similarity scoring and cluster analysis

  • Graph Database: Neo4j or similar for relationship analysis and community detection algorithms

Resilience Patterns:

  • Circuit breaker patterns prevent cascading failures when dependent services are slow or unavailable

  • Eventual consistency model allows identity resolution to proceed with cached data when real-time services are impacted

  • Fallback to simpler algorithms when advanced ML services are unavailable, with confidence scoring adjusted accordingly

Semantic Translation Dependencies

Semantic Translation Service operates with complex dependencies on ontology management systems and reasoning engines while providing critical translation capabilities that other services require for cross-context evidence sharing.

Upstream Dependencies:

  • Ontology Management System: Version-controlled storage of OWL ontologies and semantic mappings

  • Reasoning Engine Infrastructure: HermiT reasoner and OWL API for logical inference operations

  • Identity Resolution Service: Identity cluster information for context-aware translation

Downstream Dependencies:

  • Award Calculation Service: Consumes translated evidence for policy-specific entitlement calculations

  • External Integration Gateway: Provides translated evidence for partner organization consumption

  • Investigation Queue Service: Identifies semantic conflicts that require human review

Ontology Dependencies:

  • DCS Core Ontology: Base vocabularies and semantic relationships for all internal translations

  • Product-Specific Ontologies: BasicSupport, FamilyAssist, AgeCare vocabularies with regular updates

  • External Partner Ontologies: Revenue Authority, local authority, and healthcare provider vocabulary mappings

Performance Optimization:

  • Translation Cache: Redis cluster for frequently requested semantic translations with confidence preservation

  • Ontology Cache: In-memory caching of active ontology components with lazy loading for less common translations

  • Reasoning Result Cache: Pre-computed common inference patterns to reduce real-time reasoning overhead

Award Calculation Dependencies

Award Calculation Service represents the most complex dependency scenario, requiring high-quality input from multiple upstream services while maintaining strict accuracy and audit requirements for financial decisions affecting citizens.

Critical Upstream Dependencies:

  • Identity Resolution Service: High-confidence identity clusters required before award calculations can proceed

  • Semantic Translation Service: Translated evidence appropriate for specific benefit policy contexts

  • Policy Rule Engine: Drools rule sets with current rates, thresholds, and eligibility criteria

Supporting Dependencies:

  • Investigation Queue Service: Case creation when confidence thresholds are not met for automated processing

  • Notification Service: Citizen communication about award decisions, changes, and required evidence

  • External Integration Gateway: Status updates to partner systems about award outcomes

Data Quality Dependencies:

  • Confidence Threshold Configuration: Real-time access to current confidence requirements for different benefit types and award amounts

  • Historical Data Services: Access to previous award calculations for retrospective change processing

  • Rate Table Services: Current benefit rates, income thresholds, and deduction parameters

Fault Tolerance Mechanisms:

  • Graceful Degradation: Reduced automation with increased human review when confidence services are impacted

  • Transaction Rollback: Database transaction management ensures award calculations can be reversed if downstream processes fail

  • Decision Queue: Asynchronous processing enables award calculations to complete even when notification services are temporarily unavailable

Investigation Queue Dependencies

Investigation Queue Service operates with dependencies on all other platform services while implementing sophisticated workflow management that continues operating even when individual dependencies experience problems.

Workflow Dependencies:

  • http://Temporal.io Infrastructure: Workflow orchestration platform for complex investigation processes with state management

  • Case Assignment Engine: Integration with HR systems for caseworker skill profiles, workload tracking, and availability

  • Priority Scoring System: Real-time access to fraud detection algorithms and financial impact assessment

Integration Dependencies:

  • All Core Services: Investigation queue receives triggers from evidence ingestion, identity resolution, semantic translation, and award calculation

  • External Systems: Integration with partner organizations for evidence verification and fraud investigation coordination

  • Notification Service: Communication with citizens and caseworkers about investigation status and requirements

Monitoring Dependencies:

  • Performance Metrics: Real-time dashboards showing queue length, resolution times, and caseworker productivity

  • Escalation Management: Automated escalation based on case age, complexity, and business impact

  • Quality Assurance: Integration with audit systems for investigation outcome tracking and performance analysis

Cross-Cutting Service Dependencies

Several services provide capabilities that all other services depend on, creating shared dependency relationships that require careful management to prevent single points of failure.

Audit & Compliance Service Dependencies:

  • All Platform Services: Every service generates audit events requiring comprehensive logging

  • Elasticsearch Cluster: Scalable storage and search capabilities for audit trail management

  • Compliance Reporting: Integration with regulatory reporting systems and data protection management

Notification Service Dependencies:

  • All Business Services: Investigation queue, award calculation, and identity resolution trigger various notification types

  • External Communication Providers: Government notification services, email providers, postal services for multi-channel delivery

  • Citizen Preference Management: Integration with citizen portal systems for communication preference handling

External Integration Gateway Dependencies:

  • Kong API Gateway: Load balancing, authentication, and rate limiting for all external partner interactions

  • Partner Identity Management: Integration with external authentication systems and certificate management

  • All Internal Services: Gateway provides external access to evidence submission, status checking, and webhook capabilities

Dependency Management Strategies

Circuit Breaker Patterns: Each service implements circuit breakers that prevent cascading failures when dependencies are slow or unavailable. When a dependency fails, services fall back to cached data or reduced functionality rather than failing completely.

Bulkhead Isolation: Critical processing workflows are isolated from each other so that problems in one area don't affect unrelated operations. Evidence ingestion continues even when award calculation experiences problems.

Timeout and Retry Logic: All inter-service calls implement appropriate timeouts with exponential backoff retry strategies that prevent services from waiting indefinitely for dependencies while allowing recovery from transient failures.

Graceful Degradation: Services reduce functionality rather than failing completely when dependencies are unavailable. Identity resolution may use simpler algorithms when ML services are impacted, with confidence scores adjusted to reflect reduced capability.

Health Check Monitoring: Comprehensive health checks monitor both service availability and dependency health, enabling proactive intervention before problems affect citizen services.

Dependency Injection: Services use dependency injection patterns that enable testing with mock dependencies and support configuration changes without code modifications.

Event-Driven Resilience

Asynchronous Processing: Most service interactions occur through event streams rather than synchronous calls, enabling services to process at their own pace while maintaining overall system responsiveness.

Event Sourcing: Critical business events are stored in immutable event logs that enable system recovery and audit capabilities while supporting replay of processing workflows when services recover from failures.

Saga Patterns: Complex workflows that span multiple services use saga patterns to coordinate transactions while supporting compensation and rollback when individual steps fail.

Dead Letter Queues: Failed events are routed to dead letter queues for analysis and reprocessing, ensuring that temporary failures don't result in permanent data loss.

Performance and Scaling Dependencies

Shared Infrastructure Dependencies:

  • Triple Store Cluster: All services depend on the primary RDF database for evidence storage and retrieval

  • Redis Cache Cluster: Shared caching infrastructure that improves performance across all services

  • Kafka Event Infrastructure: Event streaming backbone that enables loose coupling while maintaining reliable processing

Independent Scaling: Services scale independently based on their specific performance characteristics and load patterns, with evidence ingestion scaling for document processing volume while identity resolution scales for clustering computation requirements.

Load Balancing: Each service operates behind load balancers that distribute traffic and provide health checking, enabling services to scale horizontally without affecting dependent services.

Database Connection Pooling: Shared database connections are managed through connection pools that prevent individual services from exhausting database capacity while providing fair access to shared resources.
